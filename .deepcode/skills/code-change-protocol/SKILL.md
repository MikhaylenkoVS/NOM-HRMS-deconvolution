# Code Change Protocol

Этот skill применяется ко всем диалогам в проекте. Он обязывает AI-ассистента соблюдать следующий протокол при любых изменениях кода:

## 1. План изменений

Перед тем как вносить любые изменения в код, необходимо:

- **Предложить план** изменений с явным перечислением шагов (в виде маркированного списка).
- **Обосновать каждый шаг** — зачем он нужен, что изменится, какие риски/побочные эффекты возможны.
- Если план тривиален (1-2 строки), краткое обоснование всё равно обязательно.

## 2. Запрос подтверждения

В **том же самом ответе**, где предложен план, необходимо:

- Явно спросить пользователя: вносить ли изменения?
- Явно спросить: есть ли уточнения или правки к плану?
- Не начинать выполнение автоматически — дождаться ответа пользователя.

## 3. Коммит подзадач

Как только подзадача (один логический блок изменений) завершена:

- Выполнить `git add` и `git commit` с осмысленным сообщением на русском или английском языке.
- Если подзадача не последняя — уведомить, что всё готово к следующей, и опять запросить подтверждение.

## 4. Защита критических веток (строгое правило)

**Ни при каких обстоятельствах** нельзя вносить изменения в следующие ветки без явного согласования:

| Ветка | Правило |
|-------|---------|
| `main` | ❌ **Категорически запрещено** трогать без явного согласования с пользователем. Даже `git checkout main`, `git merge`, `git push` в `main` — только после письменного подтверждения. |
| `stable` | 🚫 **Абсолютный запрет**. Эту ветку пользователь меняет **исключительно вручную**. Ассистент не имеет права выполнять никакие git-операции, затрагивающие `stable`. |

### Что считается нарушением

- `git checkout main` / `git checkout stable`
- `git merge <что-либо> в main/stable`
- `git push` в `main` / `stable`
- `git commit` на `main` / `stable`
- Любое редактирование файлов, когда `HEAD` указывает на `main` или `stable`

### Что делать вместо этого

Все изменения выполняются в **feature-ветках**, ответвляемых от `main` **только после согласования**. После одобрения пользователем слияние в `main` производится только по его явной команде.

### Проверка перед любой git-операцией

Прежде чем выполнить любую git-команду, которая затрагивает ветку, необходимо:
1. Проверить текущую ветку: `git branch --show-current`
2. Если это `main` или `stable` — **остановиться** и запросить инструкции у пользователя.

## Исключения

- Исправление опечаток по прямой просьбе пользователя может выполняться без полного плана, но коммит обязателен.
- Если пользователь явно сказал «делай без вопросов» — протокол приостанавливается до конца текущей задачи.
- Правила защиты веток (раздел 4) действуют **даже при включённом режиме «без вопросов»**.
## 5. Синхронизация с remote (автоматический пулл)

После `git push` в feature-ветку (включая `dev`) **всегда** проверять, не появились ли новые коммиты в remote:

- Если `git push` отклонился (`rejected`): выполнить `git pull --rebase`, разрешить конфликты, дозапушить.
- После успешного пуша: **перед началом следующей подзадачи** выполнить `git fetch && git status`, чтобы убедиться, что локальная ветка не отстала от remote.
- При расхождении: `git pull --rebase` перед продолжением работы.

## 6. Финальная синхронизация и отчёт (шаг по окончании запроса)

По завершении **всех** задач в рамках запроса пользователя необходимо:

### 6.1 Синхронизация remote

- Выполнить `git fetch --all --prune` для получения актуального состояния всех веток с GitHub.
- Для каждой feature-ветки, в которой велась работа, выполнить `git push` (если ещё не запушено).
- Убедиться, что `git status` не показывает расхождений с remote.

### 6.2 Отчёт

Вывести сводку в формате:

```
📋 Отчёт по сессии
Ветка: <имя ветки>
Remote: <синхронизирована / расходится / не запушена>
Коммиты (локальные):
  abc1234 краткое описание
  def5678 краткое описание
Файлы: N изменено, M добавлено, K удалено
```

- Если работа велась в нескольких ветках — вывести отчёт по каждой.
- Если все изменения уже в remote — отметить «✅ Ветка синхронизирована с remote».
- Если есть незапушенные коммиты — отметить «⚠️ Требуется push» и явно спросить пользователя.
